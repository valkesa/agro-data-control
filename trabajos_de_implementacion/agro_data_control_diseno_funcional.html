Resumen Ejecutivo

El proyecto tiene una base sólida: la jerarquía tenants/{tenantId}/sites/{siteId} ya está en Firestore, los roles owner / tenant_admin / tenant_operator ya existen en el código, y hay lógica de membresía funcional. Sin embargo, hay ocho contradicciones concretas en el código actual que deben resolverse antes de agregar tenants reales.

Riesgo principal
El mayor riesgo no es el diseño nuevo: es que el sistema ya tiene decisiones tomadas inconsistentemente entre el backend Dart, las Firestore Rules y el frontend Flutter. Agregar tenants sobre esa base sin corregirlas primero amplificará los problemas.

El MVP realista no es "armar tenants desde cero" — es cerrar las inconsistencias existentes y agregar los tres datos faltantes: nombre visible de sala, nombre visible de PLC, y la capacidad de tenant_admin de gestionar sus operadores.

2

Riesgos y Contradicciones Detectadas

C1 — Rol admin fantasma en Firestore Rules CRÍTICO
En firestore.rules, la función canEditTenantConfig permite:

memberRole(tenantId) == 'owner' || memberRole(tenantId) == 'admin'

Pero en el código Dart (UserAppRole, UserManagementService) no existe el rol admin. Nadie puede tener role='admin' hoy. En la práctica, canEditTenantConfig solo funciona para owner. Si se agrega tenant_admin pensando que hereda esta regla, no funcionará.
C2 — Visibilidad de PLCs en dos lugares MEDIO
showMunters1/showMunters2 está guardada en:
• users/{uid}.ui.visibleUnits — preferencia personal del usuario
• tenants/{tenantId}/sites/{siteId}/settings/controlDashboard.ui.visibleUnits — configuración del sitio

No hay política definida sobre cuál tiene precedencia. Producirá comportamientos inconsistentes entre usuarios del mismo tenant.
C3 — defaultSiteId hardcodeado como constante CRÍTICO para multi-site
FirestorePaths.defaultSiteId = 'genetica-1' está hardcodeado en el código Dart. Si un tenant tiene múltiples sites, el routing a Firestore se rompe silenciosamente.
C4 — URL del backend hardcodeada, sin routing por site CRÍTICO para multi-site
PlcDashboardService.fetchSnapshot() llama a una URL fija. Cada proceso backend sirve exactamente un site. El frontend no tiene lógica para seleccionar backend por tenant/site.
C5 — Rol duplicado con potencial de desincronización MEDIO
El campo role existe en users/{uid}.role y en tenants/{tenantId}/members/{uid}.role simultáneamente. Si se modifican por separado (ej: desde la consola de Firebase), el sistema se comportará diferente en cada capa.
C6 — tenant_admin no puede gestionar accesos BLOQUEANTE FUNCIONAL
Las Firestore Rules deniegan explícitamente create/update/delete en tenants/{tenantId}/members/{uid} para todos los roles. No existe ninguna Cloud Function ni endpoint para que tenant_admin asigne tenant_operator.
C7 — Nombres visibles de PLCs no existen en Firestore FALTANTE
"Munters1", "Munters2", "M1", "M2" están implícitos en la UI. No hay campo de nombre configurable en Firestore. La separación ID técnico / nombre visible no está implementada.
C8 — Selector de sala no existe FALTANTE
"Sala Genética" no viene de Firestore — viene del campo siteName del backend/config/default.json. El frontend no tiene selector de sala ni lee los documentos tenants/{tenantId}/sites/.

Propuesta de Modelo de Roles

Pregunta 1: ¿owner y admin son lo mismo?

Respuesta propuesta: Sí, unificarlos. El rol admin en las Firestore Rules es un error de diseño incompleto que nunca se implementó en Dart.

Decisión recomendada
Eliminar admin del modelo. Solo owner tiene privilegios globales. Si en el futuro se necesita un "co-owner", puede agregarse sin romper nada.

Pregunta 2: ¿Roles globales, por tenant, o mixtos?

Respuesta propuesta: Mixtos, pero con capas claras sin mezclarlas.

CAPA GLOBAL  (users/{uid}.role)
  owner           → acceso total al sistema, todos los tenants
  pending         → usuario sin asignar (hoy es 'tenant_operator' sin tenant, confuso)

CAPA POR TENANT  (tenants/{tenantId}/members/{uid}.role)
  tenant_admin    → administra su tenant: asigna operadores, configura salas
  tenant_operator → opera su tenant: ve datos, ajusta rangos si tiene permiso
Hipótesis a confirmar (→ Decisión D1)
¿Puede un usuario ser tenant_admin en un tenant y tenant_operator en otro simultáneamente? Si sí, el modelo actual (un solo activeTenantId por usuario) no lo soporta.

Pregunta 3: ¿Cómo modela tenant_admin la gestión de operadores?

  • Opción A — Permisos directos en Firestore Rules: tenant_admin puede crear/actualizar members/{uid}, validando que el rol asignado sea siempre ≤ tenant_operator.
  • Opción B — Cloud Function como intermediario: más seguro y con auditoría, pero requiere infraestructura adicional.
Recomendación para MVP
Opción A. La regla sería: tenant_admin puede crear/actualizar members/{uid} si el role a asignar es tenant_operator (nunca tenant_admin ni owner).
4

Propuesta de Modelo de Permisos

Pregunta 4: ¿Permisos por rol o feature flag por usuario?

Respuesta: por rol como base, con overrides por usuario solo para features de UI.

Permisos estructurales (leer/escribir datos sensibles):  SIEMPRE por rol en Firestore Rules
Permisos de visibilidad de UI (qué paneles aparecen):   por rol como default,
  con posibilidad de override por tenant en settings/controlDashboard.permissions
Regla de seguridad fundamental
Los feature flags solo son útiles para personalización de UX. No son un mecanismo de seguridad. Los permisos de datos sensibles siempre deben validarse en Firestore Rules o backend.

Pregunta 8: ¿Cómo ocultar botones/menús sin depender solo de frontend?

La regla es: la UI oculta, el backend/Firestore deniega. Para cada feature sensible:

  1. La UI oculta el elemento si el usuario no tiene permiso (UX)
  2. Firestore Rules deniegan la escritura si el usuario intenta igualmente (seguridad)
  3. No existe "poder editar desde frontend pero sin reglas" — son siempre las dos capas

Decisiones que Deben Cerrarse Antes de Codificar

  • D1 CRÍTICO
    ¿Un usuario puede pertenecer a múltiples tenants?
    El modelo actual tiene un único activeTenantId. Si la respuesta es "sí", hay que repensar el flujo de login y selección de contexto.
  • D2 CRÍTICO
    ¿tenant_admin puede asignar otros tenant_admin?
    ¿O solo puede gestionar tenant_operator? Si puede asignar co-admins, hay que definir quién otorga el primer tenant_admin de un tenant (propuesta: solo owner).
  • D3 IMPORTANTE
    ¿Los permisos configurables son por tenant o por usuario?
    La matriz propone overrides en settings/controlDashboard.permissions.tenant_operator. Si el requisito es por usuario específico, el override debe estar en el membership doc.
  • D4 BLOQUEANTE para multi-site
    ¿Cómo selecciona el frontend el backend correcto al cambiar de sala?
    Propuesta: agregar backendUrl al doc sites/{siteId}. ¿Es aceptable que esa URL sea visible para los miembros?
  • D5 UX
    ¿El selector de sala es un dropdown o una pantalla separada?
    Si hay pocas salas (2–3), dropdown es suficiente. Si pueden ser muchas, se necesita pantalla de selección con búsqueda.
  • D6 UX
    ¿columnLabel y displayName son el mismo campo o distintos?
    Hoy "M1" (columna) podría ser diferente a "Munters Norte" (nombre largo). ¿Configurables independientemente?
  • D7 SEGURIDAD
    ¿Quién puede leer backendUrl del site?
    Si está en sites/{siteId} accesible a miembros, la URL queda expuesta a cualquier tenant_operator.
  • D8 SEGURIDAD
    ¿"Interfaces viejas" y "Gestión de usuarios" tienen validación en Firestore?
    Si son solo UI, cualquier usuario puede acceder manipulando la app.

Modelo de Roles Resultante

Capa Global — users/{uid}.role
owner

Acceso total al sistema y todos los tenants. Configurado directamente en Firestore, nunca asignable desde la app.

pending

Usuario registrado pero sin asignación de tenant. Estado inicial de todo usuario nuevo.

Capa de Tenant — members/{uid}.role
tenant_admin

Administra su tenant: asigna operadores, configura salas y thresholds. Solo asignable por owner.

tenant_operator

Opera el sistema: ve datos y ejecuta acciones según permisos del tenant. Asignable por tenant_admin u owner.

Separación de responsabilidades
users/{uid}.role determina acceso global (solo importa si es owner). Los permisos dentro de un tenant se leen exclusivamente de members/{uid}.role. El campo users/{uid}.role también almacena el rol de tenant para display en la UI de gestión de usuarios — nunca para decisiones de seguridad.

    && (memberRole(tenantId) == 'owner' || memberRole(tenantId) == 'admin');

// DESPUÉS
function canEditTenantConfig(tenantId) {
  return isTenantAdmin(tenantId);  // delega al nuevo helper
}
firestore.rules Nuevas funciones helper Agregadas 
// Verifica que el caller sea tenant_admin activo del tenant dado
function isTenantAdmin(tenantId) {
  return isTenantMember(tenantId) && memberRole(tenantId) == 'tenant_admin';
}

// Anti-escalación: tenant_admin solo puede escribir role='tenant_operator'
// y no puede modificar su propio registro
function canTenantAdminWriteMember(tenantId, targetUid) {
  return isTenantAdmin(tenantId)
    && request.auth.uid != targetUid
    && request.resource.data.role == 'tenant_operator';
}
firestore.rules Reglas members/{uid} C6 — Bloqueante funcional resuelto 
// ANTES: todo bloqueado para todos
allow create, update, delete: if false;

// DESPUÉS
allow get:  if (signedIn() && request.auth.uid == uid && isUserActiveTenant(tenantId))
             || isOwner() || isTenantAdmin(tenantId);
allow list: if isOwner() || isTenantAdmin(tenantId);

// owner: irrestricto | tenant_admin: solo puede crear tenant_operator
allow create: if isOwner() || canTenantAdminWriteMember(tenantId, uid);

// Para update: resource.data.role verifica el rol EXISTENTE antes del cambio
// Bloquea que tenant_admin pueda modificar a otro tenant_admin
allow update: if isOwner()
  || (canTenantAdminWriteMember(tenantId, uid) && resource.data.role == 'tenant_operator');

allow delete: if false;
firestore.rules Regla users/{uid} — create y update Corregidas 
// create: nuevos usuarios arrancan con 'pending' (no 'tenant_operator')
allow create: if signedIn()
  && request.auth.uid == uid
  && request.resource.data.role == 'pending'   // antes: 'tenant_operator'
  && request.resource.data.active == false;

// update para owner: expandido de hasOnly(['role']) a todos los campos
// que escribe updateUserAccess() — antes era demasiado restrictivo
allow update: if (isOwner()
    && request.resource.data.diff(resource.data).affectedKeys()
        .hasOnly(['role', 'activeTenantId', 'defaultSiteId', 'allowedSiteIds',
                  'active', 'updatedAt', 'email']))
  || isOwnUiUpdate(uid);

Validación de Casos de Prueba

CASO 1 ✓
owner crea tenant_admin

Rules: isOwner() habilita members.create sin restricción de rol.
Servicio: updateUserAccess(role:'tenant_admin') pasa la validación (no es 'owner').

CASO 2 ✓
tenant_admin crea tenant_operator

Rules: canTenantAdminWriteMember pasa: caller es admin, targetUid ≠ callerUid, role='tenant_operator'.
Servicio: assignOperatorToTenant() pasa _assertCanAssignOperator('tenant_admin').

CASO 3 ✗ DENEGADO
tenant_admin intenta crear otro tenant_admin

Rules: canTenantAdminWriteMember falla → request.resource.data.role == 'tenant_operator' es false.
Servicio: si llama updateUserAccess(role:'tenant_admin'), no hay bloqueo en ese método (es para owner). Pero las rules lo deniegan antes de escribir. Ver advertencia A1

CASO 4 ✗ DENEGADO
tenant_admin intenta auto-elevarse (modify own record)

Rules: canTenantAdminWriteMember falla → request.auth.uid != targetUid es false.
Servicio: _assertNotSelf(callerUid, targetUid) lanza StateError antes de llegar a Firestore.

CASO 5 ✗ DENEGADO
tenant_operator intenta modificar members

Rules: isTenantAdmin(tenantId) retorna false → tanto create como update denegados.
Servicio: _assertCanAssignOperator('tenant_operator') lanza StateError inmediatamente.

CASO 6 ✓
owner puede todo

Rules: isOwner() está en todas las reglas de create/update como path alternativo sin restricciones.
Servicio: updateUserAccess() solo bloquea role == 'owner' (correcto: no auto-asignar owner desde la app).

Objetivo y Alcance

Permitir que un usuario con acceso a múltiples salas (sites) pueda cambiar entre ellas desde el frontend, y que cada sala use su propio backend Dart sin modificar la lógica interna del PLC ni las reglas de seguridad de Etapa 1.

Principio rector
Un usuario con una sola sala ve exactamente lo mismo que antes. El selector de sala solo aparece cuando hay más de una disponible. La URL del backend se obtiene de Firestore; si no existe, se cae al URL de entorno (fallback silencioso).

Fuera de alcance en esta etapa

  • Nombres visibles de PLC (Etapa 3)
  • Permisos configurables por sala
  • UI compleja de administración de sites
  • Multi-tenant por usuario

Regla de Seguridad: isOwnSiteSwitch

// Permite al usuario cambiar su propio defaultSiteId
// SOLO si el nuevo siteId ya está en su allowedSiteIds existente.
// Impide auto-otorgarse acceso a sites no autorizados.

function isOwnSiteSwitch(uid) {
  return signedIn()
    && request.auth.uid == uid
    && request.resource.data.diff(resource.data)
         .affectedKeys().hasOnly(['defaultSiteId'])
    && resource.data.allowedSiteIds.hasAny(
         [request.resource.data.defaultSiteId]
       );
}
Por qué es seguro
La validación usa resource.data.allowedSiteIds (el documento existente en Firestore), no el request.resource.data. Esto significa que el usuario no puede modificar su propia lista de sites permitidos para luego autorizarse un site nuevo — la lista la controla únicamente el owner o tenant_admin vía updateUserAccess().

Síntomas Reportados

Al abrir cualquiera de las puertas monitoreadas, el frontend mostraba la alarma de puerta abierta (ícono) pero ninguno de los cuatro comportamientos esperados ocurría:

#SíntomaComportamiento esperado
1 La tarjeta de puerta no se mostraba en rojo Fondo rojo + label "ABIERTA" mientras la puerta está abierta
2 La apertura no se registraba en Firestore Documento en doors/{doorId}/openings/{openingId} creado al abrir
3 El contador de tiempo abierta no aparecía (aunque en algún momento sí apareció) Contador dinámico con formato Xm Ys mientras la puerta está abierta

2

Arquitectura del Módulo de Tracking

El módulo de aperturas de puertas es código completamente nuevo (archivos sin commit previo). Su flujo de datos es:

// Cada ~5 s
_refreshOnce()  // en SnapshotRuntime
  │
  ├─ Poll Modbus TCP → unitsJson  // señales booleanas de PLC
  │
  ├─ initializeIfNeeded(unitsJson)  // solo la primera vez
  │     └─ _initialize()             // recovery: compara Firestore vs PLC
  │
  ├─ ingestSnapshot(unitsJson)      // detecta transiciones → escribe Firestore
  │
  └─ snapshotSummaryJson()          // estado en memoria → JSON del endpoint HTTP
        │
        └─ Frontend lee /api/snapshot cada 5 s → colorea tarjeta + muestra contador

Firestore también mantiene el estado persistido en tenants/{tenantId}/sites/{siteId}/doors/{doorId}. El frontend escucha ese documento en tiempo real (stream) para el historial.

Clave de diseño
El campo isOpen que colorea la tarjeta en rojo viene del snapshot HTTP (snapshotEvent?.isOpen), no de Firestore. Por eso, si _initialized nunca se establece en true, el snapshot siempre devuelve isOpen: false y la puerta nunca aparece roja.
3

Causa Raíz #1 — Loop Infinito de Inicialización

El problema

El método _initialize() en DoorOpeningsTracker ejecuta una fase de recovery al arrancar el backend: lee el último estado persistido en Firestore y lo compara con la señal PLC actual. Si detecta discrepancias (puerta huérfana abierta o cerrada), escribe a Firestore para normalizar el estado.

El código original no tenía ningún bloque try-catch alrededor de las tres llamadas de escritura a Firestore (recordOpened y recordClosed). Si cualquiera de esas escrituras fallaba (error de red, token expirado, timeout), la excepción propagaba hacia arriba a través de initializeIfNeeded() hasta el bloque catch de _refreshOnce().

ANTES (bug) backend/lib/src/door_openings_tracker.dart · _initialize() 
// Sin try-catch — cualquier fallo de Firestore escapa a _refreshOnce()
await _repository.recordOpened(
  door: door,
  openingId: openingId,
  openedAt: now,
  recoveredOpen: true,
);
// Si lo anterior lanza, nunca se llega a las líneas siguientes.
_states[door.doorId]!
  ..lastKnown = true
  ..activeOpeningId = openingId;

La cadena de fallo

  1. Backend arranca → _initialize() hace escritura de recovery → Firestore lanza error.
  2. Excepción llega a _refreshOnce(), que entra en el bloque catch.
  3. El catch reconstruye el estado usando _extractDoorEvents(_state.snapshotJson) — que extrae los valores del snapshot anterior (donde isOpen: false por defecto).
  4. _initialized sigue siendo false porque la asignación _initialized = true está en initializeIfNeeded(), después del await _initialize() que jamás completó.
  5. En el siguiente poll, se repite todo desde el paso 1 → loop infinito.
  6. ingestSnapshot() nunca se llama (está en el bloque try exitoso, después de initializeIfNeeded()).
  7. El snapshot HTTP siempre devuelve isOpen: false → tarjeta nunca roja → contador nunca visible.
Efecto cascada
Un único fallo transitorio de Firestore al arrancar bloqueaba permanentemente el módulo de aperturas durante toda la sesión del backend. El problema era invisible en los logs porque el catch de _refreshOnce() solo logueaba el error de poll — no indicaba que el tracking había quedado deshabilitado.
4

Causa Raíz #2 — Señal PLC null No Manejada

El problema

El método _extractDoorSignal() retorna null (tipo bool?) cuando la unidad PLC no está disponible en el snapshot (offline, timeout, etc.). Durante _initialize(), el valor de currentIsOpen puede ser null.

La cadena de condiciones original cubría solo tres casos con comparaciones estrictas (== false, == true). Si persisted.isOpen == true y currentIsOpen == null, ninguna de las tres condiciones era verdadera y el código caía en el else final:

ANTES (bug) backend/lib/src/door_openings_tracker.dart · _initialize() 
if (persisted.isOpen && currentIsOpen == false) {
  // Firestore=open, PLC=cerrado → cierra huérfana
} else if (!persisted.isOpen && currentIsOpen == true) {
  // Firestore=closed, PLC=abierto → crea apertura recuperada
} else if (persisted.isOpen && currentIsOpen == true) {
  // Ambos abiertos → restaurar en memoria
} else {
  // ← Caso: persisted.isOpen=true, currentIsOpen=null
  // Caía aquí. Ponía lastKnown=false aunque Firestore decía que estaba abierta.
  _states[door.doorId]!.lastKnown = false;
}

Consecuencia: si el PLC estaba offline en el momento exacto del arranque del backend y la puerta estaba abierta según Firestore, el tracker inicializaba el estado como cerrada, descartando la información persistida.

5

¿Por Qué "En Algún Momento Sí Aparecía"?

El módulo de tracking tiene dos rutas de inicialización controladas por isPersistenceEnabled:

isPersistenceEnabled = false

Llama a _seedStateFromSnapshot(). No hace escrituras a Firestore. Siempre completa sin errores. _initialized = true se setea. ingestSnapshot() funciona → el contador aparecía.

Efecto: Tracking en memoria funcionaba; Firestore no se escribía.

isPersistenceEnabled = true

Llama a _initialize(). Hace escrituras de recovery. Si cualquier escritura falla → excepción → _initialized queda false → loop infinito.

Efecto: El módulo quedaba permanentemente bloqueado.

Cuando Firestore no estaba correctamente configurado (isPersistenceEnabled = false), el módulo funcionaba parcialmente: el contador aparecía y la puerta se mostraba roja gracias al estado en memoria. Al configurar Firestore correctamente, isPersistenceEnabled pasó a true y el bug de loop infinito se activó.
