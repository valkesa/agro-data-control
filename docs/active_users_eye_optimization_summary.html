Optimizacion del modulo ActiveUsersEye

Resumen tecnico de los cambios realizados para eliminar lecturas permanentes de Firestore en el ojito de usuarios activos.

Objetivo

El modulo ActiveUsersEye se usaba como una herramienta de consulta puntual, pero mantenia listeners realtime activos mientras la aplicacion permanecia abierta. Esto generaba lecturas continuas de Firestore, especialmente en PWA instalada en escritorio o movil.

El objetivo fue pasar a un modelo bajo demanda:

  • Ojito cerrado = 0 lecturas.
  • Al abrir el panel, ejecutar una consulta inicial.
  • Mientras el panel esta abierto, refrescar con polling liviano.
  • Cerrar automaticamente el panel luego de 30 segundos.
  • No usar listeners realtime para presencia ni historial dentro del ojito.

Archivos modificados

lib/widgets/active_users_eye.dart

  • Se elimino el StreamBuilder del boton del ojito.
  • El widget cerrado ya no crea streams, timers ni consultas.
  • El badge conserva el ultimo conteo conocido sin refrescar en segundo plano.
  • El dialogo ahora se autocierra luego de 30 segundos.
  • El tab Activos hace un get() inicial y luego polling cada 15 segundos solo mientras esta activo.
  • Al cerrar el dialogo o cambiar de tab, se cancela el polling.
  • El tab Historial se convirtio a carga puntual con FutureBuilder.

lib/services/presence_service.dart

  • Se reemplazo watchActiveUsers() por fetchActiveUsers().
  • Se reemplazo watchUserSessions() por fetchUserSessions().
  • Ambos metodos usan get() y no snapshots().
  • fetchActiveUsers() filtra usuarios activos recientes usando lastSeenAt y activeThreshold.

Codigo relevante

Autocierre del dialogo

static const Duration _autoCloseDelay = Duration(seconds: 30);

@override
void initState() {
  super.initState();
  _tabController = TabController(length: 2, vsync: this);
  _tabController.addListener(_handleTabChange);
  _autoCloseTimer = Timer(_autoCloseDelay, _closeIfOpen);
}

Cancelacion de recursos del dialogo

@override
void dispose() {
  _autoCloseTimer?.cancel();
  _tabController.removeListener(_handleTabChange);
  _tabController.dispose();
  super.dispose();
}

Polling solo mientras el tab Activos esta activo

void _startPolling() {
  if (_refreshTimer != null) {
    return;
  }
  _loadUsers();
  _refreshTimer = Timer.periodic(_refreshInterval, (_) => _loadUsers());
}

void _stopPolling() {
  _refreshTimer?.cancel();
  _refreshTimer = null;
  _requestGeneration += 1;
}

Consulta Firestore bajo demanda

Future<List<ActiveUserPresence>> fetchActiveUsers({
  required String workspaceId,
  int limit = 50,
}) async {
  final QuerySnapshot<Map<String, dynamic>> snapshot =
      await _presenceCollection(workspaceId)
          .where('isOnline', isEqualTo: true)
          .orderBy('lastSeenAt', descending: true)
          .limit(limit)
          .get();

  final DateTime minLastSeen = DateTime.now().subtract(activeThreshold);
  return snapshot.docs
      .map(ActiveUserPresence.fromFirestore)
      .where((ActiveUserPresence user) {
        final DateTime? lastSeenAt = user.lastSeenAt;
        return lastSeenAt != null && lastSeenAt.isAfter(minLastSeen);
      })
      .toList(growable: false);
}

Cancelacion de timers y recursos

  • El widget principal ActiveUsersEye no mantiene streams ni timers.
  • El dialogo cancela _autoCloseTimer en dispose().
  • La vista de usuarios activos cancela _refreshTimer en dispose().
  • Al cambiar del tab Activos al tab Historial, se llama a _stopPolling().
  • _requestGeneration invalida respuestas asincronicas tardias para evitar actualizar widgets ya cerrados o inactivos.

Confirmacion: cuando el ojito esta cerrado, no quedan referencias activas, streams, providers ni timers asociados al modulo.

Estimacion de lecturas Firestore

Antes, estimado por hora 3.000 lecturas
Antes, estimado por dia 72.000 lecturas

Calculo usado para el caso anterior: 

10 usuarios online
5 owners con la app abierta
heartbeat cada 60 segundos

10 docs modificados por minuto * 5 listeners * 60 minutos = 3.000 lecturas/hora
3.000 * 24 = 72.000 lecturas/dia
Despues, ojito cerrado 0 lecturas
Apertura tipica menor a 10s 1 consulta

Con el panel abierto hasta el autocierre de 30 segundos:

  • Consulta inicial al abrir.
  • Refresco a los 15 segundos.
  • Puede haber un ultimo refresco cerca de los 30 segundos segun timing del timer.

Resultado aproximado: 2 a 3 consultas, con hasta 50 documentos por consulta. En el peor caso, 100 a 150 lecturas por apertura completa, en lugar de lecturas permanentes durante todo el dia.

Validacion

  • No quedan usos de snapshots() asociados a ActiveUsersEye ni a los metodos de presencia usados por el ojito.
  • No quedan referencias a watchActiveUsers ni watchUserSessions.
  • No quedan StreamBuilder en active_users_eye.dart.
  • Se ejecuto analisis puntual sin errores.
